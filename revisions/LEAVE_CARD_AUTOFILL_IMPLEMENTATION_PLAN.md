# Leave Card Autofill Implementation Plan (Name, Office, Entrance to Duty)

## Objective
Implement a leave-card autofill capability that populates at least:
- **Name**
- **Office**
- **Entrance to Duty**

while preserving the approved leave-card formatting/layout from the existing template/example.

---

## Scope (MVP)
### In scope
- Autofill the three header fields (Name, Office, Entrance to Duty) from existing employee/profile data.
- Preserve existing leave-card visual format (cell alignment, merged cells, fonts, borders, spacing, header layout).
- Keep current leave-card calculations/rows untouched unless required to insert these three fields.

### Out of scope (for this phase)
- Redesigning leave-card UI/structure.
- Adding new leave-card columns/metrics.
- Refactoring unrelated payroll/timekeeping workflows.

---

## Current-State Notes (Based on codebase scan)
- Payroll export and document-generation logic currently exists in:
  - `pages/admin/includes/payroll-management/actions.php`
  - `pages/staff/includes/payroll-management/actions.php`
- Existing code includes PDF generation and leave-card remarks text, but no explicit finalized leave-card template autofill entry point for the 3 header fields was confirmed during quick scan.
- PhpSpreadsheet is already present in dependencies, so a template-based spreadsheet path is viable.

---

## Proposed Technical Approach
## 1) Identify canonical data source for the 3 fields
- **Name**: use computed full name already used in payroll/person mappings.
- **Office**: use authoritative office/division string from person/assignment context.
- **Entrance to Duty**: use employee profile field (date hired/entry date equivalent).

### Rules
- Normalize Name to display format used in leave-card template (e.g., `LASTNAME, FIRSTNAME MIDDLENAME`).
- Use Office label aligned with approved terminology (Department/Division naming policy already in checklist).
- Render Entrance to Duty as consistent date format (PST policy still applies for timezone handling; date-only display should stay stable).

## 2) Use template-first rendering to preserve formatting
- Keep the official leave-card spreadsheet as a **base template**.
- Write only target cells for Name/Office/Entrance to Duty.
- Do not recreate styles programmatically unless unavoidable.

### Why
Template-first avoids format drift and guarantees visual consistency with the approved sample.

## 3) Add a field-to-cell mapping contract
Create a clear mapping layer (documented in code) for example:
- `name -> <cell>`
- `office -> <cell>`
- `entrance_to_duty -> <cell>`

This enables future updates if template coordinates change.

## 4) Add defensive fallback behavior
If any field is missing:
- Keep document generation successful.
- Use safe placeholders (`-`) and log a non-fatal warning/audit entry.

## 5) Integrate into existing generation flow
- Hook autofill in the same flow where leave-card or payroll-related exports/documents are generated.
- Ensure admin/staff paths use the same formatter/mapping utility to avoid role-based output differences.

---

## Implementation Steps
1. **Discovery pass**
   - Locate final leave-card generation endpoint/file used in runtime.
   - Confirm exact template file path and output format (xlsx/pdf/both).

2. **Data contract setup**
   - Define a compact payload object for leave-card header data:
     - `employee_name`
     - `office_name`
     - `entrance_to_duty`

3. **Template mapping utility**
   - Add a utility that receives payload + template file and writes only mapped cells.
   - Keep existing styles/merged ranges intact.

4. **Flow integration**
   - Call utility from generation action(s) for admin and staff paths.
   - Ensure same output behavior across roles.

5. **Validation + QA**
   - Validate output on records with complete and partial data.
   - Compare generated file against template visually and structurally.

6. **Docs update**
   - Add short maintainer notes for mapping and fallback rules.

---

## Acceptance Criteria
- Generated leave card auto-populates **Name**, **Office**, and **Entrance to Duty**.
- Existing layout/format is preserved (no unintended style or alignment changes).
- Missing one field does not break generation.
- Admin and staff generation paths produce consistent populated output.

---

## Test Plan
- **Case A (complete profile):** all 3 fields filled correctly.
- **Case B (missing office):** office shows fallback placeholder; generation succeeds.
- **Case C (missing entrance date):** placeholder shown; no crash.
- **Case D (format regression check):** compare generated file against template baseline.
- **Case E (cross-role):** same employee generated by admin vs staff path yields same populated header values.

---

## Risks and Mitigations
- **Risk:** Template cell coordinates change.
  - **Mitigation:** Isolate mapping constants and document them.

- **Risk:** Data source inconsistency between modules.
  - **Mitigation:** Use one canonical payload builder shared by generation flows.

- **Risk:** Date formatting mismatch for Entrance to Duty.
  - **Mitigation:** Centralize date formatter and enforce one display format.

---

## Deliverables for next implementation task
- Mapping utility for leave-card header autofill.
- Integrated calls in generation actions.
- Minimal maintainer documentation of field-to-cell map.
- QA evidence (before/after sample outputs).
